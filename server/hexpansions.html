Adding Hexpansion support

Each supported Hexpansion is a small Python class living in app/hexpansion/. It tells the game how to detect the hardware, what commands it can receive, and how to know when a command has been completed.

How it works

When a badge joins a room it reports its capabilities — a list of Hexpansions it has detected and the commands each one supports. The server uses this to assign commands only to badges that can actually carry them out.

On every poll the badge checks whether its current command has been completed and reports the result. If it has, the server clears it and issues a new one.

The base class

All Hexpansion modules extend HexpansionModule from app/hexpansion/base.py.

Attribute / methodWhat to do
FRIENDLY_NAME Class-level string. Must exactly match the name in the Hexpansion's EEPROM manifest. Used to detect presence.
COMMAND_OPTIONS Class-level list of strings. Every possible command this Hexpansion can receive. Can be overridden at instance level if commands are dynamic (see MegaDrive).
is_connected(hexpansions) Called on every scan. Return True if the hardware is present. The default implementation checks for a matching FRIENDLY_NAME in the hexpansions dict. Override only if connection status depends on something other than the hexpansion being physically plugged in.
on_button_down(event) Called for every button event on the badge. If the event matches the current command, set self.last_status = CommandStatus.PASSED.
check_command() Called every poll cycle. Return CommandStatus.PASSED, WAITING, or FAILED. Default returns self.last_status — fine for button-based Hexpansions; override for sensor polling (GPS, shake).
get_capabilities() Returns {"module": FRIENDLY_NAME, "commands": COMMAND_OPTIONS}. Override if the hexpansion's capabilities depend on something other than being plugged in — for example, if commands are only available after acquiring a sensor fix, or if the method needs to perform background work (e.g. draining a UART buffer) each poll cycle.
set_command(command) Called by the app when a new assignment arrives. The base implementation validates the command against COMMAND_OPTIONS and resets last_status to WAITING. Override to do per-command setup (e.g. snapshot a position, arm a sensor). Always call super().set_command(command) first.
reset() Called when the Hexpansion disconnects, or between rounds. Clear any hardware state here. Call super().reset() first.

The hexpansions dict

is_connected receives a dict keyed by port number (1–6):

# hexpansions passed to is_connected()
{
    1: {"known": True,  "name": "MegaDrive", ...},
    2: {"known": False, ...},  # connected but EEPROM unrecognised
    # unpopulated ports are absent
}

The default is_connected implementation checks item["known"] and item["name"] == self.FRIENDLY_NAME, which is correct for anything with a standard EEPROM manifest.

Writing a new module

Here's a minimal example — a hypothetical Hexpansion with a single big red button:

from .base import HexpansionModule, CommandStatus


class BigRedButtonModule(HexpansionModule):
    FRIENDLY_NAME = "BigRedButton"  # must match EEPROM name exactly
    COMMAND_OPTIONS = ["press"]

    def on_button_down(self, event):
        button = getattr(event, "button", None)
        if button is None:
            return
        if button.group != "BigRedButton":
            return
        if self.current_command == "press":
            self.last_status = CommandStatus.PASSED

Because button presses set self.last_status directly, check_command doesn't need overriding — the inherited version returns self.last_status as-is.

Sensor-based commands (like GPS or shake) can't rely on on_button_down. Override check_command instead and read your hardware there. It's called on every poll (~1 s), so keep it fast — avoid blocking I/O.

Overriding get_capabilities

Override get_capabilities when the hexpansion's reported capabilities depend on something other than being plugged in. Two real examples:

  • MegaDrive — expands from 8 to 12 commands when a 6-button pad is detected. It updates self.COMMAND_OPTIONS on the instance and the base get_capabilities picks it up automatically, so no override is needed in that case.
  • GPS — reports no commands until it has a fix, then advertises move 5m away. It overrides get_capabilities to drain the UART buffer each poll cycle so it keeps acquiring a fix in the background even before any command is assigned.

The updated list is sent to the server on the next poll so it can start (or stop) issuing those commands.

Existing modules

Three modules ship with the game. These are the best reference when writing your own.

Tildagon 2024
app/hexpansion/Tildagon2024.py

The badge itself — always connected. Buttons a–f plus shake. Drops shake from its capabilities when any hexpansion is plugged in to avoid accidental triggers. Overrides set_command to initialise the accelerometer baseline when a shake command is issued.

MegaDrive
app/hexpansion/MegaDrive.py

Sega controller hexpansion. Starts with 8 commands (3-button pad). Upgrades to 12 commands when a 6-button pad is detected via a button press — updates self.COMMAND_OPTIONS on the instance so the base get_capabilities picks it up automatically.

GPS
app/hexpansion/GPS.py

GPS hexpansion with one command: move 5m away. Reports no commands until it has a UART fix. Overrides get_capabilities to drain the UART buffer in the background, and set_command to snapshot the starting position.

Registering your module

  1. 1 Save your class as app/hexpansion/MyHexpansion.py.
  2. 2 Open app/hexpansion/__init__.py and add your import and class to MODULE_TYPES: 
from .MyHexpansion import MyHexpansionModule

MODULE_TYPES = [
    Tildagon2024Module,
    MegaDriveModule,
    GPSModule,
    MyHexpansionModule,  # add here
]
FRIENDLY_NAME must exactly match the EEPROM. The name comes from the Hexpansion's firmware manifest — check the hexpansion-firmwares repo or run scripts/generate_hexpansion_names.py to regenerate the lookup table after adding new firmwares.

Testing on hardware

Copy the app to the badge and reset it:

pipx run mpremote cp -r app/* :/apps/spaceteam/ && pipx run mpremote reset

Then, with your Hexpansion plugged in:

  1. Open the Tildateam app on the badge.
  2. From the main menu, select Test modules.
  3. The badge steps through every command for every detected Hexpansion, one at a time — the module name and command are shown on screen.
  4. Perform the action. The badge detects it and advances automatically.
  5. To skip a command, hold Cancel for 2 seconds.
  6. At the end, a summary shows how many passed and how many were skipped.

If your Hexpansion doesn't appear, it likely wasn't detected — double-check that FRIENDLY_NAME matches the EEPROM exactly and that is_connected returns True for it.
